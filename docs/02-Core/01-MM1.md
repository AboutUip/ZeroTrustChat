# MM1 安全内存模型技术规范

## 一、概述

MM1 是 ZChatIM 系统的安全核心，负责所有敏感数据的内存存储与处理。由 C++ 实现，仅通过 JNI 与 Spring Boot 交互。

**核心原则**（**产品安全取向**：**持久化扩大泄露面，须严格收紧**；**早期内存模型**下的目标表述见下；**与当前 MM2 差异**见 **一点五.1**）：
- 消息不持久化，断电即失
- 宁可销毁不泄露
- 消息分散存储 + 独立加密

> **与当前 C++ 实现**：即时消息等 **持久化由 [MM2 实现状态](../../ZChatIM/docs/Implementation-Status.md) 第2节** 承担；上列原则与「是否落盘」若与其它文档冲突，以 **`docs/AUTHORITY.md`** 及 **`ZChatIM/docs/Implementation-Status.md`** 为准。

详见 [01-Overview.md](../01-Architecture/01-Overview.md)

**与 MM2 的关系（实现口径）**：
- **MM1**：JNI 信任边界上的校验、会话、策略与安全门面（内存/RNG/密钥子模块、各 **Manager**）。
- **`MM1::EmergencyTrustedZoneWipe`**：清 **MM2** 盘态（若已初始化）与 **MM1** 列出的进程内表（见 **`JniSecurityPolicy.h` 第8条**）。**本方法不**修改 **`JniBridge::m_initialized`**。**`SystemControl::EmergencyWipe`** 在 **`EmergencyTrustedZoneWipe`** 之后会 **`JniBridge::NotifyExternalTrustedZoneWipeHandled`**，与同进程 **`JniBridge::ValidateJniCall`** 等标志对齐。**`JniInterface::EmergencyWipe`** / **`Cleanup`** 仍为主路径之一。与其它线程 **`JniBridge`** 业务并发时须**进程级互斥**（**`Get*Manager()`** 返回引用后不持 **`MM1` 锁**，见 **`JniSecurityPolicy.h` 第5节、第8节**）。
- **MM2**：消息/元数据 **SQLite + `.zdb` 编排**，并持有 **`mm2::Crypto::Init` / `Cleanup` 的全局生命周期**（`MM1::Cleanup` **不**调用 `Crypto::Cleanup`）。
- **依赖边界**：MM1 允许直接依赖 **`mm2::Crypto`** 及文档规定的 **Manager → `MM2::…`** 路由（如 **`MessageReplyManager` → `StoreMessageReplyRelation`**）；持久化业务 **禁止**在 JniBridge 绕过 MM1 直调 MM2（见 `ZChatIM/docs/JNI-API-Documentation.md`）。
- **实现进度汇总**（与代码同步维护）：[**Implementation-Status**](../../ZChatIM/docs/Implementation-Status.md) **第3节 MM1**。

**已实现 C++ 管理器（节选）**：`AuthSessionManager`（**`TryGetSessionUserId`**、**`ClearAllSessions`**、**`VerifyCredential`** 强度见 **`02-Auth.md` 第7.1节**；**`EmergencyTrustedZoneWipe` / `JniBridge::EmergencyWipe`** 清空会话及列出的 MM1 进程内态）→ [02-Auth.md](../03-Business/02-Auth.md) 第七节；`SessionActivityManager` → [04-Session.md](../03-Business/04-Session.md) 第七节；**`MessageReplyManager`**（**`TryGetSessionUserId`** 会话绑定 + **`common::Ed25519VerifyDetached`** canonical 验签：**OpenSSL** + MM2；**群友图**仍上层）→ 见 **与 MM2 的关系**、**Implementation-Status 第3节**、**`ZChatIM/docs/JNI-API-Documentation.md`**。

## 一点五、愿景与当前实现对照（读第二节前请先读本节）

### 一点五.1 第一节三条原则与现状

三条原则来自「敏感内容优先在内存」的旧威胁模型。（1）**不持久化**：减磁盘取证面；代价是本地无历史则依赖服务端。（2）**销毁优先**：紧急路径上仍有 `EmergencyTrustedZoneWipe` / `CleanupAllData` 等；第六节 L1–L3 大量仍为愿景。（3）**分散+独立加密**：愿景是堆上随机块 + Address Mapper；**当前** IM 为 **ImRam** 内逐条 **AES-GCM**（独立 `message_id`/nonce），**非**双棘轮；文件与群数据在 **`.zdb`+`data_blocks`**，**不是**第三节那种进程内堆模型。

**与实现**：IM 默认**仅 RAM、进程结束即失**；元库无 IM 表——见 [`AUTHORITY.md`](../AUTHORITY.md)、[`03-Storage.md`](03-Storage.md) 第2.6节、第七节。

**裁决**：运行行为以 **`ZChatIM/` 源码**、[`Implementation-Status`](../../ZChatIM/docs/Implementation-Status.md)、`03-Storage` 第七节为准。本文第二～五节 Random Allocator、Address Mapper、纯内存块链、双棘轮等为 **M3 愿景**，非当前 JNI 默认交付。

**泄露面 / 销毁**：[`03-Storage.md`](03-Storage.md) 第4.4节。

### 一点五.2 对照表

| 愿景小节 | 内容摘要 | **当前默认实现（权威：Implementation-Status + 源码）** |
|----------|----------|----------------------------------------|

## 三、核心组件（愿景规格）

> **说明**：**3.1～3.4** 描述的是**与当前默认 MM2 RAM IM（`StoreMessage` 不落盘）不同的**进一步内存/协议架构；**是否实现、何时实现**见 **一点五** 与 **M3 路线图**。勿将本节当作 **`ZChatIM` 已交付能力**清单。

### 3.1 Random Allocator

每次分配随机地址，物理不连续。

### 3.2 Address Mapper

加密索引：MessageID → {Address, KeySalt}

### 3.3 Key Manager

双棘轮密钥链：
```
IdentityKey → SessionKey → MessageKey(每条消息独立)
```

### 3.4 Message Manager

- 消息块独立加密
- 链式哈希（可选）
- 序列号排序

## 四、消息块结构（愿景参考布局）

> **当前 MM2**：单条 IM 为 **AES-GCM** 包（**`nonce ‖ ciphertext ‖ tag`**），**默认仅驻留 RAM**（**`ImRamMessageRow.blob`**）；**不等价**于下文字段级 **BlockHead + Content** 布局。**`.zdb` 物理格式**（文件分片等）以 **`04-ZdbBinaryLayout.md`** 为准。

```
┌────────────────────────────────────────────────────┐
│  BlockHead (16B) │ CryptoKey(32B) │ Content │Auth  │
│                  │                │ (变长)   │(16B) │
└────────────────────────────────────────────────────┘

BlockHead:
┌─────────┬──────┬───────┬───────┬────────────┐
│Magic(2) │Ver(1)│Flag(1)│Size(4)│IDHash(8)   │
└─────────┴──────┴───────┴───────┴────────────┘

Content (明文，加密存储):
┌──────────┬──────────┬──────────┬──────────┬──────────┐
│Sequence  │Timestamp │ PrevHash │ SenderID │ Payload  │
│  (8B)    │  (8B)    │  (32B)   │  (16B)   │ (变长)    │
└──────────┴──────────┴──────────┴──────────┴──────────┘
```

## 五、存储流程（愿景：纯内存路径）

> **当前路径**：**MM2 `StoreMessage` / `StoreMessages`**（见 **`MM2.h`**、**Implementation-Status 第2.1节**）。

1. 生成/获取消息密钥
2. 构造消息块 (Sequence + Timestamp + Payload)
3. 随机地址分配
4. 独立加密 (AES-256-GCM + 独立 Nonce)
5. 写入内存
6. 更新加密索引

## 六、安全机制

> **愿景与代码**：下表为**目标能力**；**与源码差距**见 **一点五.2**「第六节」「第七节」行。

### 6.1 防 Dump

| 机制 | 说明 |
|------|------|
| 内存加密 | 存储时为密文 |
| 随机地址 | 每次运行地址不同 |
| 分散存储 | 消息物理不连续 |
| 反调试 | 检测调试器，触发自毁 |

### 6.2 自毁触发

| 场景 | 动作 |
|------|------|
| 用户登出 | Level 2 擦除 |
| 会话结束 | Level 2 擦除密钥 |
| 检测调试 | Level 3 物理销毁 |
| 收到异常信号 | Level 3 全量清零 |

### 6.3 销毁级别

| 级别 | 触发场景 | 覆写轮数 | 方法 |
|------|----------|:--------:|------|
| Level 1 | 常规内存释放 | 0 | 标记释放 |
| Level 2 | 登出/会话结束 | 1 | 固定值覆写 (0x00) |
| Level 3 | 调试检测/异常信号 | 3 | 随机数覆写 + mlock |

#### mlock 保护

- Level 3 必须使用 mlock 锁定内存
- 防止敏感数据被 swap 到磁盘
- 进程终止后自动释放

## 七、侧信道防护

> **愿景与代码**：**7.1** 与部分比较逻辑在 **`common::Memory` / `SideChannel`** 中有对接；**7.2～7.4** 多为**强化目标**，**未**全量落地（见 **一点五.2**）。

### 7.1 常量时间实现

所有密码学比较必须使用常量时间算法。

### 7.2 Cache 攻击防护

| 攻击类型 | 防护措施 |
|----------|----------|
| Flush+Reload | 使用 clflush 指令清除敏感数据 |
| Evict+Reload | 避免在敏感操作中使用分支预测 |
| Prime+Probe | 使用常量时间查表实现 |

### 7.3 功耗分析防护

| 措施 | 说明 |
|------|------|
| 随机时钟插入 | 在密码学运算中插入随机延迟 |
| 功耗均衡 | 确保持续电流消耗 |
| 毛刺检测 | 检测异常电源波动 |

### 7.4 完整防护层

```
L1: 常量时间实现
    - 密钥比较
    - 签名验证
    - HMAC 计算

L2: Cache 隔离
    - 敏感数据立即清除
    - 常量时间查表
    - 内存屏障

L3: 运行时检测
    - 调试器检测
    - 虚拟机检测
    - 代码完整性校验
```

## 十一、容量估算

> **前提**：下表按**纯内存消息缓存 + 本文愿景堆模型**粗算；**当前持久化客户端**以**磁盘配额、`.zdb` 文件数、元库大小**为主，**不**直接套用下列「仅 RAM」结论。

| 项目 | 估算 |
|------|------|
| 100万在线用户 | ~80 MB |
| 5000万消息缓存 | ~3 GB |
| 密钥存储 | ~64 MB |
| 总量 | ~4 GB |
