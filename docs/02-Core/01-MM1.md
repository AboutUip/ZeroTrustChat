# MM1 安全内存模型技术规范

## 一、概述

MM1 是 ZChatIM 系统的安全核心，负责所有敏感数据的内存存储与处理。由 C++ 实现，仅通过 JNI 与 Spring Boot 交互。

**核心原则**（产品/内存模型目标）：
- 消息不持久化，断电即失
- 宁可销毁不泄露
- 消息分散存储 + 独立加密

> **与当前 C++ 实现**：即时消息等 **持久化由 [MM2 实现状态](05-ZChatIM-Implementation-Status.md) §2** 承担；上列原则与「是否落盘」若与其它文档冲突，以 **`docs/README.md`**「**冲突与权威**」及 **`05` 活文档** 为准。

详见 [01-Overview.md](../01-Architecture/01-Overview.md)

**与 MM2 的关系（实现口径）**：
- **MM1**：JNI 信任边界上的校验、会话、策略与安全门面（内存/RNG/密钥子模块、各 **Manager**）。
- **MM2**：消息/元数据 **SQLite + `.zdb` 编排**，并持有 **`mm2::Crypto::Init` / `Cleanup` 的全局生命周期**（`MM1::Cleanup` **不**调用 `Crypto::Cleanup`）。
- **依赖边界**：MM1 允许直接依赖 **`mm2::Crypto`** 及文档规定的 **Manager → `MM2::…`** 路由（如 **`MessageReplyManager` → `StoreMessageReplyRelation`**）；持久化业务 **禁止**在 JniBridge 绕过 MM1 直调 MM2（见 `ZChatIM/docs/JNI-API-Documentation.md`）。
- **实现进度汇总**（与代码同步维护）：[05-ZChatIM-Implementation-Status.md](05-ZChatIM-Implementation-Status.md) **§3 MM1**。

**已实现 C++ 管理器（节选）**：`AuthSessionManager`（含 **`TryGetSessionUserId`**）→ [02-Auth.md](../03-Business/02-Auth.md) 第七节；`SessionActivityManager` → [04-Session.md](../03-Business/04-Session.md) 第七节；**`MessageReplyManager`**（**`TryGetSessionUserId`** 会话绑定 + **`common::Ed25519VerifyDetached`** canonical 验签：**OpenSSL** + MM2；**群友图**仍上层）→ 见 **§与 MM2 的关系**、**`05` §3**、**`ZChatIM/docs/JNI-API-Documentation.md`**。

## 三、核心组件

### 3.1 Random Allocator

每次分配随机地址，物理不连续。

### 3.2 Address Mapper

加密索引：MessageID → {Address, KeySalt}

### 3.3 Key Manager

双棘轮密钥链：
```
IdentityKey → SessionKey → MessageKey(每条消息独立)
```

### 3.4 Message Manager

- 消息块独立加密
- 链式哈希（可选）
- 序列号排序

## 四、消息块结构

```
┌────────────────────────────────────────────────────┐
│  BlockHead (16B) │ CryptoKey(32B) │ Content │Auth  │
│                  │                │ (变长)   │(16B) │
└────────────────────────────────────────────────────┘

BlockHead:
┌─────────┬──────┬───────┬───────┬────────────┐
│Magic(2) │Ver(1)│Flag(1)│Size(4)│IDHash(8)   │
└─────────┴──────┴───────┴───────┴────────────┘

Content (明文，加密存储):
┌──────────┬──────────┬──────────┬──────────┬──────────┐
│Sequence  │Timestamp │ PrevHash │ SenderID │ Payload  │
│  (8B)    │  (8B)    │  (32B)   │  (16B)   │ (变长)    │
└──────────┴──────────┴──────────┴──────────┴──────────┘
```

## 五、存储流程

1. 生成/获取消息密钥
2. 构造消息块 (Sequence + Timestamp + Payload)
3. 随机地址分配
4. 独立加密 (AES-256-GCM + 独立 Nonce)
5. 写入内存
6. 更新加密索引

## 六、安全机制

### 6.1 防 Dump

| 机制 | 说明 |
|------|------|
| 内存加密 | 存储时为密文 |
| 随机地址 | 每次运行地址不同 |
| 分散存储 | 消息物理不连续 |
| 反调试 | 检测调试器，触发自毁 |

### 6.2 自毁触发

| 场景 | 动作 |
|------|------|
| 用户登出 | Level 2 擦除 |
| 会话结束 | Level 2 擦除密钥 |
| 检测调试 | Level 3 物理销毁 |
| 收到异常信号 | Level 3 全量清零 |

### 6.3 销毁级别

| 级别 | 触发场景 | 覆写轮数 | 方法 |
|------|----------|:--------:|------|
| Level 1 | 常规内存释放 | 0 | 标记释放 |
| Level 2 | 登出/会话结束 | 1 | 固定值覆写 (0x00) |
| Level 3 | 调试检测/异常信号 | 3 | 随机数覆写 + mlock |

#### mlock 保护

- Level 3 必须使用 mlock 锁定内存
- 防止敏感数据被 swap 到磁盘
- 进程终止后自动释放

## 七、侧信道防护

### 7.1 常量时间实现

所有密码学比较必须使用常量时间算法。

### 7.2 Cache 攻击防护

| 攻击类型 | 防护措施 |
|----------|----------|
| Flush+Reload | 使用 clflush 指令清除敏感数据 |
| Evict+Reload | 避免在敏感操作中使用分支预测 |
| Prime+Probe | 使用常量时间查表实现 |

### 7.3 功耗分析防护

| 措施 | 说明 |
|------|------|
| 随机时钟插入 | 在密码学运算中插入随机延迟 |
| 功耗均衡 | 确保持续电流消耗 |
| 毛刺检测 | 检测异常电源波动 |

### 7.4 完整防护层

```
L1: 常量时间实现
    - 密钥比较
    - 签名验证
    - HMAC 计算

L2: Cache 隔离
    - 敏感数据立即清除
    - 常量时间查表
    - 内存屏障

L3: 运行时检测
    - 调试器检测
    - 虚拟机检测
    - 代码完整性校验
```
