# MM1 安全内存模型技术规范

## 一、概述

MM1 是 ZChatIM 系统的安全核心，负责所有敏感数据的内存存储与处理。由 C++ 实现，仅通过 JNI 与 Spring Boot 交互。

**核心原则**（**产品安全取向**：**持久化扩大泄露面，须严格收紧**；**早期内存模型**下的目标表述见下；**与当前 MM2 差异**见 **一点五.1**）：
- 消息不持久化，断电即失
- 宁可销毁不泄露
- 消息分散存储 + 独立加密

> **与当前 C++ 实现**：即时消息等 **持久化由 [MM2 实现状态](05-ZChatIM-Implementation-Status.md) 第2节** 承担；上列原则与「是否落盘」若与其它文档冲突，以 **`docs/README.md`**「**冲突与权威**」及 **`05` 活文档** 为准。

详见 [01-Overview.md](../01-Architecture/01-Overview.md)

**与 MM2 的关系（实现口径）**：
- **MM1**：JNI 信任边界上的校验、会话、策略与安全门面（内存/RNG/密钥子模块、各 **Manager**）。
- **`MM1::EmergencyTrustedZoneWipe`**：清 **MM2** 盘态（若已初始化）与 **MM1** 列出的进程内表（见 **`JniSecurityPolicy.h` 第8条**）。**若进程使用 `JniBridge`**，紧急擦除须走 **`JniInterface::EmergencyWipe`**（或 **`Cleanup`**）；**勿单独**调用 **`MM1::GetSystemControl().EmergencyWipe()`**，否则 **`JniBridge::m_initialized`** 可能与已清空的 **MM2** 不一致。
- **MM2**：消息/元数据 **SQLite + `.zdb` 编排**，并持有 **`mm2::Crypto::Init` / `Cleanup` 的全局生命周期**（`MM1::Cleanup` **不**调用 `Crypto::Cleanup`）。
- **依赖边界**：MM1 允许直接依赖 **`mm2::Crypto`** 及文档规定的 **Manager → `MM2::…`** 路由（如 **`MessageReplyManager` → `StoreMessageReplyRelation`**）；持久化业务 **禁止**在 JniBridge 绕过 MM1 直调 MM2（见 `ZChatIM/docs/JNI-API-Documentation.md`）。
- **实现进度汇总**（与代码同步维护）：[05-ZChatIM-Implementation-Status.md](05-ZChatIM-Implementation-Status.md) **第3节 MM1**。

**已实现 C++ 管理器（节选）**：`AuthSessionManager`（**`TryGetSessionUserId`**、**`ClearAllSessions`**、**`VerifyCredential`** 强度见 **`02-Auth.md` 第7.1节**；**`EmergencyTrustedZoneWipe` / `JniBridge::EmergencyWipe`** 清空会话及列出的 MM1 进程内态）→ [02-Auth.md](../03-Business/02-Auth.md) 第七节；`SessionActivityManager` → [04-Session.md](../03-Business/04-Session.md) 第七节；**`MessageReplyManager`**（**`TryGetSessionUserId`** 会话绑定 + **`common::Ed25519VerifyDetached`** canonical 验签：**OpenSSL** + MM2；**群友图**仍上层）→ 见 **与 MM2 的关系**、**`05` 第3节**、**`ZChatIM/docs/JNI-API-Documentation.md`**。

## 一点五、愿景与当前实现对照（读第二节起正文前请先读本节）

### 一点五.1 为何仍保留「消息不持久化」等原则（早期内存模型在说什么）

**第一节**三条原则**不是**随手写的口号，而是同一套 **「敏感内容主要活在进程内存里」** 的威胁模型下的推论；文档成文时，产品假设更接近 **「终端 = 可信执行环境 + RAM 缓存」**，而不是 **「终端 = 带本地数据库的 IM 客户端」**。

#### （1）消息不持久化，断电即失

- **意图**：降低 **设备丢失 / 被扣押 / 冷启动取证** 时，从**磁盘**直接拖出明文或整库聊天记录的风险；同时简化「卸载 / 登出 = 数据自然消失」的产品叙事。
- **代价**：无网、杀进程、换机即**不能**靠本地找回历史——必须由**服务端同步**或用户显式备份承担可用性。
- **与当前实现**：**本仓库默认**走 **MM2**，IM **落 `.zdb` + 元库**，**重启后仍在**（除非 **`Cleanup` / `CleanupAllData` / 删目录**）。这是**显式产品取舍**，不是实现疏漏。原则行仍保留，是为了标明 **「曾考虑的威胁面」** 与 **「当前未选该取舍」**。

#### （2）宁可销毁不泄露

- **意图**：在 **认证失败、调试器、异常信号、用户登出** 等场景下，优先 **擦除密钥与明文缓存**，而不是优先「尽量保住会话」。
- **与当前实现**：**紧急擦除**（**`EmergencyTrustedZoneWipe` / `CleanupAllData`**）与 **MM1 进程内表清空** 仍体现这一取向；**分级销毁（第六节 L1–L3）** 里大量条目仍是**愿景**，未按表全量产品化。

#### （3）消息分散存储 + 独立加密

- **「分散」**：在早期模型里通常指 **堆上多块、不连续、与 MessageID 的映射不暴露在单一连续 blob**，使 **整段 dump / 单文件拷贝** 难以一次性还原全量会话；必要时配合 **Random Allocator / Address Mapper**（见第三节）。
- **「独立加密」**：每条消息（或每块）使用 **独立 nonce/密钥材料**，避免 **一条密钥泄漏 → 整盘历史解密**；与是否落盘**正交**——落盘也可以每条独立 **AES-GCM**，但**不再**等于「纯内存分散」。
- **与当前实现**：**MM2** 对每条消息有 **独立 `message_id`、独立密文包（nonce‖ciphertext‖tag）**，**密钥派生链**与 **双棘轮** **不是一回事**；**物理分散**体现在 **多 `.zdb` 文件、块索引在 SQLite**，**不是**早期文档里的 **进程内随机地址堆**。

**裁决**（与 **`docs/README.md` 第2节「冲突与权威」**一致）：**运行行为**以 **`ZChatIM/` 源码**、**[05-ZChatIM-Implementation-Status.md](05-ZChatIM-Implementation-Status.md)**、**[03-Storage.md](03-Storage.md) 第七节** 为准；**本文第二节至第五节**中 **Random Allocator、Address Mapper、纯内存消息块、双棘轮链** 等为 **阶段 M3 可选架构愿景**，**不是**当前 **`ZChatIM` / JNI 默认交付路径**。

**持久化带来的泄露面与销毁路径**（与「内存模型」冲突时的**安全讨论**）：见 **`03-Storage.md` 第4.4节**（**密钥+目录一并失守**时的边界、**`CleanupAllData` / 紧急擦除**、**登出是否清盘**须产品显式调用等）。

## 四、消息块结构（愿景参考布局）

> **当前 MM2**：单条 IM 为 **AES-GCM** 包（**`nonce ‖ ciphertext ‖ tag`**），**默认仅驻留 RAM**（**`ImRamMessageRow.blob`**）；**不等价**于下文字段级 **BlockHead + Content** 布局。**`.zdb` 物理格式**（文件分片等）以 **`04-ZdbBinaryLayout.md`** 为准。

```
┌────────────────────────────────────────────────────┐
│  BlockHead (16B) │ CryptoKey(32B) │ Content │Auth  │
│                  │                │ (变长)   │(16B) │
└────────────────────────────────────────────────────┘

BlockHead:
┌─────────┬──────┬───────┬───────┬────────────┐
│Magic(2) │Ver(1)│Flag(1)│Size(4)│IDHash(8)   │
└─────────┴──────┴───────┴───────┴────────────┘

Content (明文，加密存储):
┌──────────┬──────────┬──────────┬──────────┬──────────┐
│Sequence  │Timestamp │ PrevHash │ SenderID │ Payload  │
│  (8B)    │  (8B)    │  (32B)   │  (16B)   │ (变长)    │
└──────────┴──────────┴──────────┴──────────┴──────────┘
```

## 五、存储流程（愿景：纯内存路径）

> **当前路径**：**MM2 `StoreMessage` / `StoreMessages`**（见 **`MM2.h`**、**`05` 第2.1节**）。

1. 生成/获取消息密钥
2. 构造消息块 (Sequence + Timestamp + Payload)
3. 随机地址分配
4. 独立加密 (AES-256-GCM + 独立 Nonce)
5. 写入内存
6. 更新加密索引

## 六、安全机制

> **愿景与代码**：下表为**目标能力**；**与源码差距**见 **一点五.2**「第六节」「第七节」行。

### 6.1 防 Dump

| 机制 | 说明 |
|------|------|
| 内存加密 | 存储时为密文 |
| 随机地址 | 每次运行地址不同 |
| 分散存储 | 消息物理不连续 |
| 反调试 | 检测调试器，触发自毁 |

### 6.2 自毁触发

| 场景 | 动作 |
|------|------|
| 用户登出 | Level 2 擦除 |
| 会话结束 | Level 2 擦除密钥 |
| 检测调试 | Level 3 物理销毁 |
| 收到异常信号 | Level 3 全量清零 |

### 6.3 销毁级别

| 级别 | 触发场景 | 覆写轮数 | 方法 |
|------|----------|:--------:|------|
| Level 1 | 常规内存释放 | 0 | 标记释放 |
| Level 2 | 登出/会话结束 | 1 | 固定值覆写 (0x00) |
| Level 3 | 调试检测/异常信号 | 3 | 随机数覆写 + mlock |

#### mlock 保护

- Level 3 必须使用 mlock 锁定内存
- 防止敏感数据被 swap 到磁盘
- 进程终止后自动释放

## 七、侧信道防护

> **愿景与代码**：**7.1** 与部分比较逻辑在 **`common::Memory` / `SideChannel`** 中有对接；**7.2～7.4** 多为**强化目标**，**未**全量落地（见 **一点五.2**）。

### 7.1 常量时间实现

所有密码学比较必须使用常量时间算法。

### 7.2 Cache 攻击防护

| 攻击类型 | 防护措施 |
|----------|----------|
| Flush+Reload | 使用 clflush 指令清除敏感数据 |
| Evict+Reload | 避免在敏感操作中使用分支预测 |
| Prime+Probe | 使用常量时间查表实现 |

### 7.3 功耗分析防护

| 措施 | 说明 |
|------|------|
| 随机时钟插入 | 在密码学运算中插入随机延迟 |
| 功耗均衡 | 确保持续电流消耗 |
| 毛刺检测 | 检测异常电源波动 |

### 7.4 完整防护层

```
L1: 常量时间实现
    - 密钥比较
    - 签名验证
    - HMAC 计算

L2: Cache 隔离
    - 敏感数据立即清除
    - 常量时间查表
    - 内存屏障

L3: 运行时检测
    - 调试器检测
    - 虚拟机检测
    - 代码完整性校验
```

## 十一、容量估算

> **前提**：下表按**纯内存消息缓存 + 本文愿景堆模型**粗算；**当前持久化客户端**以**磁盘配额、`.zdb` 文件数、元库大小**为主，**不**直接套用下列「仅 RAM」结论。

| 项目 | 估算 |
|------|------|
| 100万在线用户 | ~80 MB |
| 5000万消息缓存 | ~3 GB |
| 密钥存储 | ~64 MB |
| 总量 | ~4 GB |
