## 三、存储机制

> **实现权威**：MM2 编排（**`Initialize` / `StoreFileChunk` / `StoreMessage` / `StoreMessages` / `GetSessionMessages` / `MessageQueryManager::List*` / `DeleteMessage` 等**）、**SQLite 元数据路径**（**默认 SQLCipher**，**`ZCHATIM_USE_SQLCIPHER`**，见 **`03-Storage.md` §4.2**）、**`mm2_message_key.bin`**、**AES-GCM / PBKDF2 / 随机数**（**OpenSSL 3**）以 **`03-Storage.md` 第七节** 与源码 **`MM2.h` / `MM2.cpp` / `Crypto.h` / `Crypto.cpp`** 为准；本节为概念与历史结构说明，数字与流程若与 第七节 / **`04-ZdbBinaryLayout.md`** 冲突以 第七节与二进制文档为准。

### 3.1 .zdb 文件

```
目录: data/
文件: [随机名].zdb（单文件容量 5MB，与 Types.h 中 ZDB_FILE_SIZE 一致）
数量: Types.h 定义 ZDB_MIN_FILES=5、ZDB_MAX_FILES=2000；当前 ZdbManager 未强制卷数上下界（见 03-Storage.md 第七节）
单次写入上限: ≤500KB（ZDB_MAX_WRITE_SIZE）
填充: 创建时 payload 区预填高熵字节（04-ZdbBinaryLayout.md；AllocateSpace 等预留区写 0）
```

> **与旧版文档对齐**：单文件 **5MB**，不是 500KB；500KB 为**单次写入**上限（见 `01-Overview.md` 存储表）。

### 3.2 数据存储

```
写入流程（概念）:
1. 选择 .zdb 分卷（实现见 ZdbManager::SelectFile）
2. v1：在 usedSize 处尾部追加（非「随机洞位」）；见 04-ZdbBinaryLayout.md
3. 写入载荷（分片明文或 StoreMessage 密文包）
4. 记录到 SQLite（zdb_files / data_blocks 等）
```

> **注意**：早期「随机空闲位置」叙述与 **v1 仅尾部追加** 不一致；以 **`04-ZdbBinaryLayout.md`** 第1节 / 第3节 为准。

### 3.3 并发控制

```
方案（目标/产品叙述）: 文件锁 + SQLite 事务
- 写操作: 加写锁 + BEGIN EXCLUSIVE
- 读操作: 加读锁
- 粒度: 每个文件独立锁
```

> **与当前 C++ 差异**：`ZdbManager` 用 **`m_mutex` 串行化**公开 API；`SqliteMetadataDb` 为 **逐语句**写入、**未**统一包在 **`BEGIN EXCLUSIVE`** 事务里，也**未**实现「每 .zdb 独立文件锁」的完整方案。以 **`03-Storage.md` 第七节** 与源码为准。

## 四、数据持久化

### 4.1 持久化数据

| 数据类型 | 存储位置 |
|----------|----------|
| 用户元数据 | .zdb |
| 好友列表 | .zdb |
| 群组信息 | .zdb |
| 文件密钥 | .zdb (加密) |
| 聊天消息 | **`.zdb`（密文包）+ SQLite `im_messages`/`data_blocks`**（**`StoreMessage`**；见 **`03-Storage.md` 第二节 / 第七节**） |
| 会话密钥 | MM1/协议层内存（轮换见 **`05-KeyRotate.md`**）；与 MM2 **消息加密密钥文件** `mm2_message_key.bin` **不同** |

### 4.2 SQLite 表

字段以 **`docs/02-Core/03-Storage.md`** 与源码 **`SqliteMetadataDb::kCreateSchema`** 为准（实现中 **无** `block_id` 列；块由 **`data_id` + `chunk_idx`** 唯一标识）。

```text
zdb_files: file_id, total_size, used_size
data_blocks: data_id, chunk_idx, file_id, offset, length, sha256
user_data: user_id, file_id, offset, length, sha256, type
group_data: group_id, file_id, offset, length, sha256
group_members: group_id, user_id, role, joined_at
im_messages: message_id, session_id
```

**`group_members.role`**：**0 / 1 / 2**（见 **`03-Storage.md` §2.5**）；**MM1 `GroupManager`** 经 **`MM2::*ForMm1`** 维护。**`group_data`**：**`ZGK1`** 信封（§2.4 / **`03-Storage.md`**）。**`TryGetGroupKeyEnvelopeForMm1` / `SeedAcceptedFriendshipForSelfTest`**：**非 JNI** 契约，见 **`JniSecurityPolicy.h`**。

## 五、密钥管理

### 5.1 密钥层级

```
Master Key → 文件加密密钥 → 用户密钥
```

### 5.2 刷新机制

**轮换周期与双密钥并行**以 **[05-KeyRotate.md](../03-Business/05-KeyRotate.md)** 为权威（Identity / Master / Session / Message 分级，旧密钥保留 1 小时等）。MM2 侧流程概要：

```
流程: 生成新密钥 → 重新加密数据 → 更新存储 → 按策略销毁旧密钥
```

## 六、数据过期

| 数据类型 | 过期时间 | 处理 |
|----------|----------|------|
| 聊天消息（磁盘） | **7 天（产品策略）** | **当前 MM2 无自动按天裁剪**；可调用 **`CleanupSessionMessages`**（按会话）或运维删库（见 **`05` 第2.1节、第2.2节**） |
| 会话密钥（Session Key） | **1 小时**（轮换周期，见 05-KeyRotate 第1.1节） | 轮换/失效后需重新协商（MM1 内存语义） |
| 临时文件 | 24 小时（产品） | 标记可复用（实现依业务） |

> **冲突消除**：不得写「7 天到期 = MM2 内存销毁」。落盘消息 **仅显式删除/清理** 才消失。权威见 **`docs/README.md`**「冲突与权威」。

### 7.1 在线推送

```
用户A 发送 → 服务器 → 用户B (在线)
```

### 7.2 离线同步

## 十、服务重启

**本仓库 ZChatIM 进程重启（与 `docs/README.md`「冲突与权威」一致）**：

```
1. MM1：会话表、限流等内存态丢失；须客户端重新 Auth
2. MM2：.zdb + zchatim_metadata.db 仍在磁盘；已 StoreMessage / StoreFileChunk 的数据保留
3. 消息密钥文件 mm2_message_key.bin：仍在 indexDir（除非 CleanupAllData）
4. 重建：运行时索引与解密能力随 Initialize + 首次读路径恢复
```

**错误叙述（已废弃）**：~~「重启丢弃消息/文件」~~ — 与 **v1 落盘实现** 冲突。
