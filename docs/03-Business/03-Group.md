# 群密钥分发机制技术规范

## 一、密钥层级

```
Identity Key (用户长期密钥)
    │
    ▼
Group Key (群共享密钥) ← MM1 管理，24小时刷新
    │
    ▼
Member Key (成员加密密钥) → .zdb 存储 (公钥加密)
```

## 二、密钥存储

| 密钥类型 | 存储位置 | 容量 |
|----------|----------|------|
| Identity Key | MM1 内存 | 用户级 |
| Group Key | MM1 内存 + .zdb 备份 | 群级 |
| Member Key | .zdb | 每成员 128B |

## 三、.zdb 分片规则

- 文件大小: 5MB（**`ZDB_FILE_SIZE`**）
- 单次写入: ≤500KB（**`ZDB_MAX_WRITE_SIZE`**）
- **v1 物理写入**：各 `.zdb` 卷 **尾部追加**（非随机洞位），见 **`docs/02-Core/04-ZdbBinaryLayout.md`**
- 存储结构（产品）: 按成员 ID 等逻辑分片；与 MM2 **`data_blocks`/`file_id`** 索引配合

## 四、加密流程

```
发送:
1. MM1 获取 Group Key
2. AES-256-GCM 加密消息 (独立Nonce)
3. 发送密文

接收:
1. .zdb 获取 Member Key
2. Identity Key 解密 Member Key → Group Key
3. Group Key 解密消息
```

## 五、轮换机制

| 触发条件 | 范围 | 操作 |
|----------|------|------|
| 24小时定时 | 全局 | 所有群 Group Key 刷新 |
| 成员加入 | 单群 | 新密钥分发 → 等待确认 |
| 成员退出 | 单群 | 旧密钥销毁 + 新密钥分发 |
| 手动触发 | 单群 | 强制刷新 |
| 群主注销 | 单群 | 强制转让群主 |

## 六、分发确认机制

```
群主/管理员分发密钥:
1. 生成新 Group Key
2. 加密分发给每位成员
3. 成员收到 → 验证 → 返回 ACK
4. 发送方记录确认状态
5. 等待超时 → 重新发送

成员未确认:
- 限制: 无法发送/接收群消息
- 重试: 最多3次
- 超时: 标记为异常成员
```

## 七、成员管理

### 加入流程

```
新成员加入:
1. 收到 GROUP_INVITE
2. 同意 → 加入群
3. 获取 Group Key (需确认)
4. 确认成功 → 正常收发消息
```

### 踢出流程

```
成员被踢:
1. 群主/管理员发送 GROUP_REMOVE
2. MM1 生成新 Group Key
3. 移除被踢成员 Member Key
4. 分发给其余成员 (需确认)
5. 被踢成员: 标记为"可重新申请"
```

### 重新申请

```
被踢成员重新申请:
1. 发送 GROUP_JOIN_REQUEST
2. 群主/管理员审核
3. 通过 → 重新分发 Group Key
4. 拒绝 → 保持禁止状态
```

### 群主转让 (上线确认)

```
群主转让:
1. 当前群主发送 GROUP_TRANSFER_OWNER
2. 指定新群主用户ID
3. 验证新群主是否为成员
4. 标记待确认 → 等待新群主上线确认

新群主不在线:
- 转让标记为"待确认"
- 新群主下次上线时确认
- 确认后生效，旧群主降为普通成员

新群主拒绝:
- 转让取消，保持原群主
```

### 群主注销

```
群主注销时:
1. 检测群主账户状态为"已注销"
2. 自动触发 GROUP_TRANSFER_OWNER
3. 转让给下一位管理员
4. 无管理员 → 转让给最老成员
5. 转让失败 → 解散群

安全:
- 群主注销后无法操作群
- 成员无法知道群主是否注销
- 群数据保留 (加密状态)
```

## 八、群成员列表加密

```
存储结构:
- 成员列表加密后存 .zdb
- 成员ID + 角色 + 加入时间
- 加密密钥: Group Key

访问控制:
- 仅群主/管理员可查看完整列表
- 普通成员: 不可查看
- 物理获取: 无法解析
```

## 九、安全保证

- Group Key: MM1 可信区存储，Level 3 销毁
- Member Key: .zdb 加密存储
- 分发确认: 双向 ACK
- 传输: **ZSP** 承载 + 端到端加密；链路层 **TLS** 为部署可选项（见 **`docs/01-Architecture/02-ZSP-Protocol.md` 第十节**）
- 校验: SHA-256 完整性验证

## 十、与当前 ZChatIM native 实现的对照（避免与上文产品愿景混淆）

| 主题 | 本文档（产品/目标） | **当前代码**（权威：`ZChatIM/docs/Implementation-Status.md`） |
|------|---------------------|--------------------------------------------------------------------------|
| 成员列表可见性 | 第八节：倾向管理员可见完整列表 | **`GetGroupMembers`**：任意**在群成员**可拉取 **`user_id` 列表**（SQLite **`group_members`**） |
| 群密钥 / `group_data` | Group Key、Member Key、**.zdb** 大块、成员级分发 | **已实现本地信封 `ZGK1`**：**`group_data` + `data_blocks` chunk0**（**44B**：魔数 + epoch + 32B 随机）；**`CreateGroup`** 写首包，**`UpdateGroupKey`** 由 **owner/admin** 轮换；**尚无** per-member 加密分发 / ACK（见 第六节） |
| 邀请 | 协议层 GROUP_INVITE | **须** **`friend_requests` status=1** 边（邀请者视角 **accepted** 列表含被邀请人） |
| JNI 权限 | 协议层 GROUP_* | **`docs/06-Appendix/01-JNI.md`** + **`mm1::GroupManager`**（**principal** 作操作者、**role** 0/1/2） |
| 踢人 / 退群后密钥 | 第五节：踢人触发轮换 | **当前**：**`RemoveMember`/`LeaveGroup`** **不**自动 **`UpdateGroupKey`**；须产品层显式轮换或后续接自动策略 |

产品收紧策略（如成员列表 ACL）须在 **MM1** 与 **第八节** 同步改版。
