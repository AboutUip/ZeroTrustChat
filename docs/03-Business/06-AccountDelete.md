# 账户注销技术规范

> **JNI**：**`deleteAccount` / `isAccountDeleted`**（**`docs/06-Appendix/01-JNI.md`** 第十节）。  
> **本地实现（ZChatIMCore）**：**`mm1::AccountDeleteManager`** — **`reauthToken` 与 `secondConfirmToken` 同长 ≥16B 且内容相同**（双确认由产品生成相同载荷或上层保证）；写 **`mm1_user_kv`** **`type=0x41434431`（'ACD1'）** 单字节墓碑 **`{1}`**；**不**自动 **`CleanupAllData`**。产品级「密钥销毁 / 好友与群状态清理」仍须在上层或扩展实现中与 **`03-Storage.md`** 叙述对齐。  
> **冲突与权威**：**`docs/README.md`**「冲突与权威」。

## 一、注销流程

```
用户发起注销:
1. 身份验证 (重新认证)
2. 确认注销 (二次验证)
3. 执行注销 (不可逆)
```

## 二、执行步骤

```
1. 验证身份:
   - 要求用户重新输入密码
   - 验证 session 有效性

2. 密钥销毁 (Level 3):
   - 销毁 MM1 中所有密钥
   - 标记用户状态为"已注销"
   - 清除好友关系
   - 清除群组成员资格

3. 确保数据不可解密:
   - 由密钥销毁保障，数据内容无法还原明文
```

## 三、安全机制

```
设计原则:
- 不删除任何数据
- 仅销毁密钥
- 数据内容不可解密（由密钥销毁保障）

验证:
- 必须重新认证
- 需要二次确认

销毁:
- Level 3 级别
- MM1 密钥覆写
- 标记"已注销"
```

## 四、服务器处理

```
收到注销请求:
1. 验证 session 和密码
2. MM1 执行 Level 3 销毁
3. 标记用户为"已注销"
4. 数据不可解密，且后续访问将按已注销状态拒绝或无法解密
```

## 五、数据处理

| 数据类型 | 处理方式 | 说明 |
|----------|----------|------|
| 密钥 | Level 3 销毁 | 物理覆写 |
| 好友关系 | 标记清除 | 逻辑删除 |
| 群组成员 | 标记清除 | 逻辑删除 |
| 会话 | 清除 | 终止会话与会话状态 |
| 其它加密数据 | 保留 | 密钥销毁后不可解密 |

## 六、访问控制

```
已注销用户:
- 登录: 永久禁止
- 数据访问: 无法解密
- 物理获取: 无法恢复明文
```
