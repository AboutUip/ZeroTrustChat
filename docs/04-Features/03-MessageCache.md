# 消息缓存上限技术规范

> **冲突处理（必读）**：**不得**将本页当作「当前 MM2 把消息只放内存」。**权威**见 **`docs/README.md`**「冲突与权威」、**`docs/02-Core/03-Storage.md` 第七节**、**`docs/02-Core/05-ZChatIM-Implementation-Status.md`**。  
> **事实**：IM 消息经 **`StoreMessage`** **持久化**到 **`.zdb` + SQLite**；**无**独立「100 条 LRU **内存**缓存」模块与实现一致。

---

## 一、当前实现 vs 产品目标

| 维度 | **当前 ZChatIM C++** | **产品目标（下文 第二至四节）** |
|------|----------------------|------------------------------|
| 消息存储 | **磁盘密文 + 索引** | 可叠加 **热缓存**（未单独实现） |
| 未读 LRU / 已读标记 | **`MM2::MarkMessageRead` / `GetUnreadSessionMessages` 已实现**（**`im_messages.read_at_ms`**）；**`MessageQueryManager` 未封装**（走 MM2 或后续 JNI） | 未读淘汰、已读不参与淘汰（产品层） |
| 进程重启 | **消息仍在磁盘**（除非删目录 / `CleanupAllData`） | 若仅有内存缓存则会丢——**不适用于当前客户端** |

---

## 二、产品目标：缓存策略（待实现/对接 MM1）

```
┌─────────────────────────────────────────────────────────────┐
│  消息热缓存（规划，非当前 MM2 持久化层）                        │
├─────────────────────────────────────────────────────────────┤
│  限制: 100条/会话                                            │
│  淘汰: LRU (仅未读消息)                                       │
│  过期: 7天（策略；磁盘 TTL 须另设清理任务或 API）               │
│  已读: 不参与淘汰                                             │
└─────────────────────────────────────────────────────────────┘
```

---

## 三、会话队列（规划）

```
per_session_message_queue（内存或上层模块，非 SqliteMetadataDb）
  sessionId → deque<msgId> (max 100)
  LRU 仅针对未读；已读不淘汰
```

---

## 四、过期与重启（按部署区分）

- **磁盘消息**：**不会**因「服务重启」自动消失；**7 天**仅为策略，可调用 **`MM2::CleanupSessionMessages`**（按会话清空索引与块）或运维流程。  
- **仅内存队列**：重启清空——若未来实现，须在架构图中标明 **与 MM2 磁盘并存**。

---

## 五、参数（产品 / 验收）

| 项目 | 值 | 说明 |
|------|-----|------|
| 未读上限 | 100 条/会话 | 目标；**未映射到当前 MM2 API** |
| 淘汰规则 | LRU（仅未读） | 依赖 **未读状态**；见 **05 第2.2节** |
| 过期时间 | 7 天 | **策略**；非当前自动行为 |
| **持久化存储** | **`.zdb` + SQLite** | **实际消息所在**（**03 第七节**） |
